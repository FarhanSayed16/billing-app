## 1. Critical Business Decisions

The Super Admin has provided the following constraints that fundamentally shape how the product is built.There are 4 major changes from the original PROJECT_COMPLETE_DOCUMENTATION.md:

| # | Decision | Original Plan | Revised Plan | Reason |
|---|---|---|---|---|
| 1 | **Branding** | App is branded under the Super Admin's company name | App is **white-label / neutral**. Each store shows ITS OWN branding. The app has its own product name (BillPush) but stays invisible to the customer. | Super Admin distributes the app to independent store owners. If the app carried the Super Admin's company branding, it would create conflicts and confusion with franchise owners who have their own brand identity. |
| 2 | **WhatsApp Delivery** | Meta WhatsApp Cloud API (paid, automated, server-to-server) | **Native Share Intent** (free, manual trigger by employee via phone's share sheet) | Super Admin does not want to pay for any API tiers. Zero external API cost is a hard requirement. |
| 3 | **Admin Login** | Super Admin creates Store Admin accounts and sends invitations | Store Admins **register themselves** using their own email. Super Admin can then approve, assign roles, and manage them. | Simpler onboarding. Store Admins manage their own credentials. |
| 4 | **Customer Portal Auth** | OTP-based login via WhatsApp | **No login at all.** Customer enters their phone number or Billing ID and sees their invoices directly. | Lower complexity. No OTP cost. Faster customer experience. |

## 2. The White-Label Branding Model

This is the single most important architectural decision in the product.

### The Problem
The Super Admin owns the platform and distributes it to multiple independent store owners (e.g., "Three Brothers Clothing", "Fashion Hub", "Style Street"). These stores are NOT franchises of the Super Admin's brand — they are independent businesses using the BillPush platform as a tool. If the app displayed "SuperAdmin's Company Name" on every invoice, the store owners would reject it.

### The Solution: Neutral Platform, Store-Level Branding

**What the customer sees:**
- The invoice header shows the STORE's name, logo, and contact info (e.g., "Three Brothers Clothing").
- The invoice footer shows a small watermark: "Invoice generated by BillPush" with a link to the BillPush web portal.
- The customer never sees the Super Admin's company name anywhere.

**What the store admin sees:**
- When a Store Admin first logs in, they set up their Store Profile: Store Name, Store Logo, Store Address, Store Phone, GST Number.
- All invoices generated by that store's employees automatically carry that store's branding.
- The Store Admin does NOT see BillPush branding prominently — the app feels like "their own" billing tool.

**What the Super Admin sees:**
- The Super Admin sees the BillPush admin dashboard with data from ALL stores.
- They see store names, not their own branding — because each store is an independent entity.

### Customer Web Portal Branding
When a customer visits `bills.billpush.com` and looks up their bill:
- The BillPush landing page is clean and neutral (BillPush's own branding).
- Once the customer enters their number/billing ID and views a specific invoice, that invoice page renders with the STORE's branding (logo, colors, name).
- The page footer still says "Powered by BillPush".

This way:
- BillPush gets brand visibility as the platform.
- Each store gets full ownership of their invoice identity.
- Zero conflict between Super Admin and Store Admins.

## 3. WhatsApp Delivery Strategy (Zero Cost)

### The Constraint
The Super Admin will NOT pay for any WhatsApp API (Meta Cloud API, Twilio, etc.). We need a completely free solution.

### The Solution: Native Share Intent

Instead of automated server-to-server messaging, the app will use the phone's built-in **Share functionality** to send the invoice.

**How it works:**

1. Employee completes the billing process and taps "Confirm and Send".
2. The app generates the invoice PDF and saves it locally on the device.
3. The app also generates a shareable link (e.g., `bills.billpush.com/v/A1B2C3`).
4. The app opens the device's **native share sheet** (Android: Share Intent / iOS: Share Sheet).
5. The share sheet is pre-populated with:
   - **Recipient:** The customer's WhatsApp number
   - **Message:** A formatted text like:
     ```
     Thank you for shopping at Three Brothers Clothing!
     Your bill: ₹1,468
     View your invoice: bills.billpush.com/v/A1B2C3
     ```
   - **Attachment:** The invoice PDF file
6. The employee taps WhatsApp from the share options, confirms, and sends.

### What This Means

| Aspect | Paid API Approach | Share Intent Approach (Ours) |
|---|---|---|
| Cost | ₹500-5000/month for API | Completely free |
| Automation | Fully automatic, no employee action needed | Employee taps 2-3 extra buttons (Share > WhatsApp > Send) |
| Delivery tracking | We know if message was delivered/read | We do NOT know delivery status |
| Bulk campaigns | Possible | Not possible via share sheet |
| Reliability | 100% automated | Depends on employee completing the share action |

### Share Options Available to Employee
The share sheet will offer:
- **WhatsApp** (primary intended target)
- **SMS** (fallback if customer is not on WhatsApp)
- **Email** (if customer prefers email)
- **Any other app** installed on the employee's phone

### Website Link Share
In addition to the PDF, the shareable link (`bills.billpush.com/v/A1B2C3`) allows the customer to view the invoice on the web at any time. This link is permanent and never expires.

### Future Upgrade Path
If the Super Admin ever decides to pay for WhatsApp API in the future, we can add automated messaging as an OPTIONAL premium feature without changing the existing architecture. The share intent remains as the free default.

## 4. Revised Authentication and Login System

### Super Admin
- **Registration:** The Super Admin account is created during the initial app setup (first-time installation). There is only one Super Admin.
- **Login:** Email + Password.
- **Capabilities:** Can view all stores, approve/reject Store Admin registrations, assign or revoke store admin access, view all analytics and data.

### Store Admin (Changed)
- **Registration:** The Store Admin downloads the BillPush app and registers themselves using their OWN email address and password. They sign up as a "Store Manager".
- **First-Time Setup:** After registration, they create their Store Profile (Store Name, Logo, Address, GST Number, Phone).
- **Approval:** The Super Admin sees a notification that a new Store Admin has registered. The Super Admin can:
  - Approve the registration (the store becomes active).
  - Reject it (the store is not activated).
  - Later: Suspend or revoke the Store Admin's access.
- **Login:** Email + Password (their own credentials, NOT created by Super Admin).
- **Role Assignment:** The Super Admin can change a user's role (e.g., demote from admin, or promote if needed) but does NOT create the credential itself.

### Employee
- **No change from original plan.**
- Created by the Store Admin with a Name + 4-digit PIN.
- Logs in by selecting their name from a dropdown and entering the PIN.
- Can be deactivated by the Store Admin at any time.

### Why This Change?
- Simpler onboarding: Store Admins don't need to wait for an invitation from the Super Admin.
- Store Admins own their own credentials — less dependency on the Super Admin.
- Super Admin retains full control via the approval/rejection mechanism.

## 5. Customer Lookup (Simplified, No Login)

### The Constraint
No OTP, no login, no account creation for customers. Keep it dead simple.

### The Solution

**Option A: Billing ID Lookup (No login needed)**
1. Customer visits `bills.billpush.com`.
2. Enters the 16-character Billing ID found on their digital/shared invoice.
3. The specific invoice loads immediately. No login. No OTP.

**Option B: Phone Number Lookup (No login needed)**
1. Customer visits `bills.billpush.com`.
2. Enters their 10-digit phone number.
3. The system shows all invoices linked to that phone number across all stores.
4. Customer taps any invoice to view the full detail.

### Security Consideration
Since there is no OTP verification, anyone who knows a phone number could theoretically view that person's purchase history. To mitigate this:
- The invoice list shows only: Date, Store Name, and Total Amount. No sensitive details are visible in the list.
- To view the full invoice detail (itemized list), the user must enter the specific Billing ID (which only the actual customer has, from their WhatsApp message).
- This two-layer approach (phone number shows summary, Billing ID shows full detail) balances convenience and privacy.

### Revised Customer Web Portal Flow

```
bills.billpush.com
       │
       ├── "I have a Billing ID"
       │       │
       │       └── Enter 16-char Billing ID
       │               │
       │               └── Full invoice detail loads
       │                   (Store branding, items, taxes, PDF download)
       │
       └── "I want to see my purchase history"
               │
               └── Enter 10-digit phone number
                       │
                       └── List of invoices (Date, Store, Amount)
                               │
                               └── Tap any invoice
                                       │
                                       └── "Enter Billing ID to view full detail"
                                           OR
                                           "View summary only"
```

## 6. Revised Invoice Branding and Layout

Every invoice generated by the system follows this branding hierarchy:

### Header (Store's Branding)
- Store Logo (uploaded by Store Admin)
- Store Name (e.g., "Three Brothers Clothing")
- Store Address
- Store Phone Number
- Store GST Number

### Body (Transaction Details)
- Invoice Number (e.g., `TB-2026-00047` where prefix comes from store name)
- Date and Time
- Customer Name and Phone
- Itemized product list with quantities and prices
- Subtotal, Tax Breakdown (CGST + SGST), Discounts, Grand Total
- QR Code (encodes the Billing ID for quick lookup)

### Footer (Platform Attribution)
- Small text: "Invoice generated by BillPush"
- Web link: `bills.billpush.com/v/{billing_id}`
- This is the ONLY place BillPush's name appears on the customer-facing invoice.

### Key Rule
The Store Admin's phone number or the Admin's number is used as the WhatsApp sender (since the employee shares the invoice FROM the store's device/WhatsApp account). The customer sees the message coming from the store, NOT from BillPush.

## 7. Revised Operational Workflows

### Workflow A: Store Admin Onboarding (Revised)
1. Store Admin downloads the BillPush app.
2. Taps "Register as Store Manager".
3. Enters: Name, Email, Password, Phone Number.
4. Account is created in a "Pending Approval" state.
5. Super Admin receives a push notification: "New store registration: [Name] ([Email])".
6. Super Admin opens the app, reviews the request, taps "Approve".
7. Store Admin receives a notification: "Your account has been approved! Set up your store."
8. Store Admin logs in and creates their Store Profile: Store Name, Upload Logo, Address, GST Number.
9. Store Admin creates Employees: Names + 4-digit PINs.
10. Store is now fully operational.

### Workflow B: The Checkout and Invoice Sharing (Revised)
1. Employee logs in with PIN, taps "New Bill".
2. Enters customer's phone number. System auto-fills name if returning customer.
3. Scans or manually adds items to cart.
4. Taps "Confirm and Send".
5. **Behind the scenes:**
   - Invoice is saved to PostgreSQL with all details.
   - PDF is generated locally on the device.
   - A shareable web link is generated.
6. The app opens the phone's **Share Sheet**, pre-filled with:
   - Message text (store name, amount, link)
   - PDF attachment
   - Customer's WhatsApp number pre-selected (on Android via `intent` targeting)
7. Employee taps WhatsApp from the share options and hits Send.
8. Customer receives the message on WhatsApp — it appears to come from the store's WhatsApp number (whoever's phone the employee is using).

### Workflow C: Customer Checks an Old Bill (Revised)
1. Customer opens `bills.billpush.com` on any browser.
2. They enter their phone number or the Billing ID from their old WhatsApp message.
3. If phone number: they see a summary list of all their purchases. They can click any to see partial details.
4. If Billing ID: they see the full branded invoice with the store's logo, itemized list, and a PDF download button.
5. No login required. No OTP. No account creation.

## 10. Final Tech Stack

| Layer | Technology | Notes |
|---|---|---|
| **Mobile App** | Flutter (Dart) | Cross-platform (iOS + Android). Camera for barcode scanning. Native Share Intent for WhatsApp sharing. Hive/SQLite for offline caching. |
| **Web Portal** | React (Next.js 15) | Customer-facing bill lookup portal. Server-side rendering for SEO. Also serves Super Admin's web dashboard for heavy data exports. |
| **Backend API** | Node.js with NestJS | RESTful API. Handles all business logic. Serves both mobile app and web portal. |
| **Database** | PostgreSQL 16 | Relational. Multi-tenant with Row-Level Security. Prisma as ORM. |
| **File Storage** | AWS S3 or Cloudflare R2 | Stores: store logos, product images, generated invoice PDFs. |
| **PDF Generation** | Flutter pdf package (in-app) + Puppeteer (server-side for web) | PDFs are generated locally on the employee's phone for instant sharing. Server also generates for the web portal. |
| **Push Notifications** | Firebase Cloud Messaging (FCM) | Push to Super Admin and Store Admin devices (daily summaries, new registration alerts, etc.). |
| **Caching** | Redis | Cache product catalogs and customer lookups for POS speed. |
| **Hosting** | Vercel (Web) + Railway or AWS (Backend + DB) | Cost-effective. Easy deployment. |

**What is NOT in the stack anymore:**
- Meta WhatsApp Cloud API (removed)
- Twilio (removed)
- WhatsApp OTP service (removed)

## 11. Revised Database Changes

The following changes are needed on top of the original table definitions in PROJECT_COMPLETE_DOCUMENTATION.md:

### Modified: `users` table
- Add column: `approval_status` ENUM (`PENDING`, `APPROVED`, `REJECTED`, `SUSPENDED`)
- Store Admins start as `PENDING` after self-registration. Super Admin changes to `APPROVED`.

### Modified: `invoices` table
- Remove column: `whatsapp_status` (no longer trackable without paid API)
- Add column: `share_triggered` BOOLEAN (true if the employee opened the share sheet)
- Add column: `invoice_pdf_url` VARCHAR (S3/R2 URL to the generated PDF)

### Modified: `stores` table
- Add column: `logo_url` VARCHAR (store's own logo, uploaded by Store Admin)
- Add column: `phone` VARCHAR (store's contact phone, used as WhatsApp sender identity)
- Add column: `brand_color` VARCHAR (optional hex color for invoice branding)

### Removed: `campaigns` table
- Deferred entirely. Marketing campaigns require paid WhatsApp API.

### Everything else remains exactly as defined in PROJECT_COMPLETE_DOCUMENTATION.md Section 10.

## 12. Development Phases (Updated Roadmap)

### Phase 1 — Core MVP
**Goal:** A working billing app that generates invoices and shares them via WhatsApp.

- Backend API: Auth (with self-registration + approval flow), Stores CRUD, Employees CRUD, Customers CRUD, Invoices CRUD.
- Flutter Mobile App:
  - Super Admin: Approve/reject store registrations. View all stores.
  - Store Admin: Set up store profile (name, logo, address). Create employees.
  - Employee: POS screen with manual item entry. Customer phone number capture. Invoice generation. PDF generation. Native share sheet to WhatsApp.
- Web Portal: Billing ID lookup page (single invoice view, no login).
- Database: PostgreSQL with all core tables.

### Phase 2 — Intelligence
**Goal:** Analytics and catalog management.

- Super Admin: Global analytics dashboard (total revenue, store leaderboard, customer count).
- Store Admin: Store-level analytics (daily/weekly revenue, peak hours, employee performance).
- Product catalog management: Super Admin uploads catalog. Employee barcode scanning via camera.
- Customer Web Portal: Phone number lookup (purchase history summary list).
- Push notifications: Daily revenue summaries to Store Admin and Super Admin.

### Phase 3 — Engagement
**Goal:** Loyalty program and returns.

- Loyalty and rewards: Auto points earning, cross-store redemption, balance display on POS.
- Returns and exchanges: QR code scanning from customer phone, refund processing, points reversal.
- Offline mode: Full offline caching with background sync.

### Phase 4 — Scale (Future)
**Goal:** Advanced features that may require paid services.

- Inventory management with low-stock alerts.
- Audit logs and compliance exports.
- OPTIONAL: Meta WhatsApp Cloud API integration (if Super Admin decides to pay).
- OPTIONAL: WhatsApp chatbot and marketing campaigns.
- OPTIONAL: Multi-Super-Admin support (multiple brands on one platform).
